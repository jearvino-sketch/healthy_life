# 江浙本地饭店采集工作流

## 目标与范围

从 Bilibili 10 万粉以上博主发布的无锡、苏州、常州、镇江及其县级市探店视频中，整理评论区真实出现的饭店推荐，再用高德／百度地图和大众点评核对门店信息，最终形成可追溯、可复查的本地饭店清单。

本流程的核心不是照抄博主视频中的店，而是提取**评论区主动推荐的店**。所有粉丝数、评分、评价数、营业状态和地址都必须记录核验日期，因为这些信息会变化。

## 一、新电脑环境

### 1. 必需软件

- Git
- Google Chrome
- Node.js 18 或更高版本
- Codex 或其他能够调用终端的编码代理
- Playwright CLI

安装并检查 Playwright CLI：

```powershell
npm install -g @playwright/cli@latest
playwright-cli --help
playwright-cli install --skills
```

Playwright 官方资料：

- [Coding agents 安装说明](https://playwright.dev/python/docs/getting-started-cli)
- [可见浏览器、浏览器类型和配置](https://playwright.dev/agent-cli/configuration)
- [持久化会话与人工接管面板](https://playwright.dev/agent-cli/sessions)

### 2. 单独使用一个采集会话

不要连接日常使用的 Chrome 默认配置，也不要让自动化读取个人浏览记录。为本项目建立独立的持久化会话：

```powershell
playwright-cli -s=jiangzhe-restaurants open https://www.bilibili.com --browser=chrome --headed --persistent
playwright-cli show
```

说明：

- `--headed`：弹出可见 Chrome。
- `--persistent`：浏览器关闭后仍保留登录状态。
- `-s=jiangzhe-restaurants`：固定使用本项目的独立会话。
- `playwright-cli show`：打开可视化面板，遇到扫码、短信、滑块、验证码时由用户接管鼠标和键盘。

Windows 默认会把持久化配置放在 `%LOCALAPPDATA%\ms-playwright\` 下，不应放进 Git 仓库。Playwright 也明确不建议自动化直接使用日常 Chrome 的默认用户配置。

## 二、登录与通路检查

所有账号登录、扫码、短信和验证码必须由用户本人完成，不向 agent 提供密码、Cookie、短信验证码或登录态文件。

保持同一个 `jiangzhe-restaurants` 会话，依次打开并登录：

```powershell
playwright-cli -s=jiangzhe-restaurants goto https://www.bilibili.com
playwright-cli -s=jiangzhe-restaurants goto https://www.dianping.com
playwright-cli -s=jiangzhe-restaurants goto https://ditu.amap.com
playwright-cli -s=jiangzhe-restaurants goto https://map.baidu.com
```

登录检查：

1. B 站能够打开个人主页、视频完整评论区和评论回复。
2. 大众点评不再跳转“身份核实”，能够搜索店铺并打开完整评价页。
3. 高德和百度地图能够搜索门店、切换分店并查看营业状态。
4. 关闭会话后，用相同命令重新打开；若仍保持登录，说明持久化成功。

遇到验证码时：

1. 停止自动翻页和请求。
2. 执行 `playwright-cli show`。
3. 用户手动完成验证。
4. 确认页面恢复正常后再继续。

不得绕过验证码、伪造设备指纹、注入或导出 Cookie，也不得在网站提示访问过快后继续高频请求。

## 三、B 站视频筛选

### 1. 博主门槛

满足以下条件才进入正式来源表：

- 采集当天粉丝数 `≥ 100,000`。
- 视频内容明确与目标城市或其县级市的饭店／美食探店有关。
- 能确认视频发布时间、BV 号、博主 UID 和视频链接。
- 优先近三年视频；更早视频只作为线索，必须重新核验门店仍在营业。

每个来源视频记录：

| 字段 | 内容 |
| --- | --- |
| 城市／县级市 | 视频实际探店区域 |
| 博主名称 | 发布者名称 |
| 博主 UID | 用于去重和复核 |
| 粉丝数 | 采集当天页面显示值 |
| 粉丝核验日期 | `YYYY-MM-DD` |
| 视频标题 | 原标题 |
| BV 号 | 视频唯一编号 |
| 发布时间 | 页面显示日期 |
| 播放量 | 采集时显示值 |
| 视频 URL | 原始链接 |

同一博主的多个视频可以收录，但不能把它们当作多个独立博主来源。

### 2. 搜索词

按城市、县级市与下列关键词组合搜索：

```text
探店 / 美食 / 苍蝇馆 / 老字号 / 本地人推荐 / 小馆子 / 面馆 / 早餐 / 夜宵
```

县级市范围以采集当期官方行政区划为准，不把普通市辖区误写成县级市；市辖区可以作为地址字段保留。

## 四、完整评论采集

### 1. “完整评论”的操作定义

对每个入选视频：

1. 使用“按时间”或能够覆盖全部评论的排序翻到最后一页。
2. 采集所有顶层评论。
3. 展开每条评论下的全部回复，不只保留默认显示的前几条。
4. 再检查热门／置顶评论，补齐可能没有出现在当前排序中的内容。
5. 以评论 ID 去重，不能把热门排序和时间排序中的同一评论重复计数。
6. 记录实际抓取数量与页面显示评论总数；不一致时标记“未完整”，不得宣称已经抓全。

只保存研究所需字段：

| 字段 | 内容 |
| --- | --- |
| BV 号 | 评论所属视频 |
| 评论 ID | 去重与追溯 |
| 父评论 ID | 回复关系；顶层为空 |
| 发布时间 | 页面时间 |
| 点赞数 | 采集时数值 |
| 评论文本 | 原文或最小必要摘录 |
| 推荐店名 | 从评论中识别的店名 |
| 推荐菜 | 评论明确提及时记录 |
| 地点线索 | 城市、区县、街道、商场等 |
| 采集日期 | `YYYY-MM-DD` |

隐私最小化：最终清单不展示评论者昵称、头像、UID、主页或其他个人信息。原始评论只用于证明推荐确实出现，不建立用户画像。

### 2. 推荐识别规则

计为有效推荐：

- 明确出现店名，且语义为推荐、值得吃、建议博主去、当地人常去等。
- 店名不完整但包含足够地点／菜品线索，之后能唯一对应到具体门店。
- 回复中补充了店名或分店信息，可与父评论组合判断。

不计为有效推荐：

- 只说“好吃”“下次来”而没有可识别店名。
- 单纯复述视频里已经探访的店，没有新增推荐含义。
- 广告、团购引流、复制粘贴、明显刷屏或同一评论的重复回复。
- 只有菜名，没有任何能够定位饭店的线索。

同一个人重复发送相同店名只计一次；无法可靠判断是否同一人时，以评论 ID 去重，不做身份推断。

## 五、店名标准化与去重

评论中的店名先进入候选表，不立即合并：

1. 保留评论原始写法。
2. 去除“饭店、餐馆、面馆”等非必要空格和异体标点，但不擅自改名。
3. 根据城市、区县、道路、商圈和招牌照片确认正式名称。
4. 同品牌不同分店分行记录。
5. 老店／新店、总店／分店、搬迁前／搬迁后不能只凭名称合并。
6. 名称相似但地址不同且无品牌关系时，视为不同门店。

建议使用内部唯一键：

```text
城市 + 区县 + 标准店名 + 分店名或完整地址
```

## 六、地图与大众点评交叉核验

### 1. 地图核验

高德和百度地图至少核对以下字段：

- 标准店名和分店名
- 城市、区县、完整地址
- 坐标或地图分享链接
- 营业中、暂停营业、已关闭或无法确认
- 营业时间
- 联系电话（仅用于区分同名门店，最终表可不展示）
- 最近核验日期

两张地图信息冲突时，不自行选择“看起来更合理”的一个；记录冲突并交由人工复核。

### 2. 大众点评核验

记录页面当日显示值：

- 点评正式店名和分店
- 类目／菜系
- 评分
- 评价数量
- 人均消费
- 地址
- 营业状态
- 推荐菜
- 最近评价日期
- 店铺 URL
- 核验日期

评分、评价数和人均会变化，禁止只写数值而不写核验日期。若页面被验证页遮挡、字段未显示或店铺没有评分，填写“未获取”，不能猜测。

### 3. 营业状态判断

- 地图和点评均显示正常，且近期仍有评价：`营业中`。
- 任一平台明确显示永久关闭：`疑似关闭`，人工再次确认后移出推荐清单。
- 地址搬迁：保留旧址备注，主记录改为新址并注明迁址日期。
- 同名店无法确定评论指向哪家分店：`分店待确认`。

## 七、证据等级

| 等级 | 条件 |
| --- | --- |
| A | 至少 3 条去重后的有效评论推荐，地图与点评信息一致，且有近期营业证据 |
| B | 至少 2 条有效评论推荐，地图与点评均能确认门店 |
| C | 只有 1 条有效评论推荐，或分店／营业状态仍有一项待确认 |
| 排除 | 无法定位、确认关闭、纯广告、只有视频推荐而评论区无人推荐 |

评论数量只代表在入选视频样本中的出现次数，不代表全网口碑，也不能与大众点评评分直接相加。

## 八、最终饭店表结构

正式结果写入后续文件 `饮食/江浙本地饭店.md`，建议字段如下：

| 城市 | 区县／县级市 | 店名 | 分店 | 菜系 | 评论推荐数 | 推荐菜 | B站来源 | 博主粉丝数／核验日 | 地址 | 地图状态 | 点评分／评价数／人均 | 点评核验日 | 证据等级 | 备注 |
| --- | --- | --- | --- | --- | ---: | --- | --- | --- | --- | --- | --- | --- | --- | --- |

一个饭店对应一行；多个来源视频使用 `<br>` 分隔，不能为了表格简短而删除来源链接。

## 九、原始数据与安全边界

建议目录：

```text
饮食/
├─ 江浙本地饭店采集工作流.md
├─ 江浙本地饭店.md
└─ 饭店数据/
   ├─ 视频来源.csv
   ├─ 评论推荐.csv
   └─ 门店核验.csv
```

可以提交：

- 视频和店铺公开 URL
- 店名、地址、评分、人均、评价数及核验日期
- 去除个人信息后的推荐统计
- 必要且简短的评论证据摘要

禁止提交：

- Chrome／Playwright 用户配置目录
- Cookie、localStorage、sessionStorage 或认证状态 JSON
- 账号、密码、手机号、短信验证码和二维码截图
- 评论者昵称、UID、头像和主页资料
- 绕过验证码、反爬或访问限制的代码

## 十、执行节奏

按城市分批，避免一次任务过大：

1. 无锡及县级市
2. 苏州及县级市
3. 常州及县级市
4. 镇江及县级市
5. 全量去重、复核与排序

每个城市完成后执行：

1. 检查视频来源、评论数和门店数能相互追溯。
2. 抽查至少 10% 的评论与门店页面。
3. 检查 Markdown 表格列数和链接。
4. 只暂存本次任务文件。
5. 提交并推送当前分支到 `origin`。

建议每 6 个月复核一次营业状态；复核时更新日期，不覆盖历史备注。

## 十二、故障处理

| 问题 | 处理 |
| --- | --- |
| Chrome 没有弹出 | 确认命令包含 `--headed`，检查 Chrome 是否安装，执行 `playwright-cli config-print` |
| 重启后登录丢失 | 确认始终使用同一个 `-s=jiangzhe-restaurants` 和 `--persistent` |
| 出现验证码／身份核实 | 立即停止自动操作，执行 `playwright-cli show` 后人工完成 |
| 大众点评字段缺失 | 标记“未获取”，保留 URL，后续人工复核 |
| 评论数量对不上 | 检查排序、分页、折叠回复和评论 ID 去重；仍不一致时标记“未完整” |
| 地图出现多个同名店 | 根据评论地点线索、分店名和地址逐一核对，不直接合并 |
| 网站提示访问频繁 | 停止采集并降低频率，不轮换账号、代理或设备规避限制 |
